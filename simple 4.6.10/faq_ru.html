Что означают цифры в версии модуля и как часто стоит его обновлять?
Версия модуля состоит из трех цифр, разделенных точками. Первая цифра меняется тогда, когда модуль полностью переписывается. Вторая - при добавлении нового значимого функционала. Третья - при исправлении ошибок и добавлении незначительного функционала. Частота обновления зависит только от Вашего желания, так как ошибки необязательно повторяются у всех в силу разнообразия настроек модуля. Так как модуль достаточно легко обновляется через /update, то можно не переживать насчет обновлений. Если Вы нашли какую-либо ошибку в модуле, прежде всего обновитесь до последней версии, может быть данная ошибка уже решена.
Обязателен ли вкмод для установки модуля?
Нет, не обязателен. Вкмоды, которые ставятся вместе с модулем нужны только лишь для отображения кастомных полей модуля в админ части и в письмах. Они не критичны для работы модуля и их можно не ставить либо удалить после установки, при этом сами кастомные поля будут работать и сохраняться, просто не будут отображаться в админке, но значения этих полей можно получить, создав код для получения (смотри соответствующий раздел FAQа)
Нужно ли ставить вкмоды (оцмоды) модуля
Вкмоды из каталога "opencart 1.5.x - vqmods for custom fields" или оцмоды из "opencart 2.x - ocmods for custom fields" нужно ставить, если только создали кастомные поля и хотите видеть их значения в админке опенкарта (сами они могут работать и без вкмодов или оцмодов). Вкмоды (оцмоды) из каталога "usefull vqmods but not required" ставить не нужно вообще, без явного понимания, зачем они нужны.
Как отображать значения кастомных полей модуля в админке опенкарта
Если версия опенкарта 1.5.х, то нужно залить вкмоды из каталога "opencart 1.5.x - vqmods for custom fields" в /vqmod/xml/ Если версия опенкарта 2.х.х, то нужно установить оцмоды из "opencart 2.x - ocmods for custom fields" в админке опенкарта.
Как обновлять модуль?
Достаточно залить файлы из /update на сервер, не забывая заливать шаблоны из каталога /update for X.X.X/catalog/view/theme/default/template/ в свою тему /update for X.X.X/catalog/view/theme/ВАШАТЕМА/template/ Данный каталог (/update) содержит файлы "ядра" модуля, поэтому при заливке не пострадают правки шаблонов, стилей или настройки. Единственное, что может пострадать - это изменения в файле catalog/model/tool/simpleapimain.php Это АПИ основных полей опенкарта, но как правило, в нем не нужно делать правки (хотя это и допускается) Для кастомных полей модуля файл АПИ catalog/model/tool/simpleapicustom.php и при обновлении данный файл не перезаписывается, так как его нет в файлах обновления
Я обновил модуль через каталог /update, но в админ части (или в основной) вижу, что не подгружается часть текста
Каталог /update содержит только файлы "ядра" модуля, чтобы облегчить обновление модуля и предотвратить затирание пользовательских изменений в стилях, настройках, переводах. По указанной причине каталог /update не содержит языковые файлы модуля, а так как в модуль постоянно вносятся изменения и добавляется функционал, то в нем могут быть добавлены новые строки в языковые файлы. Поэтому, в данном случае можно скопировать языковые файлы модуля из каталога /upload /upload/admin/language/ в /admin/language /upload/catalog/language/ в /catalog/language При этом будьте внимательны, если до этого сами сделали изменения в языковых файлах модуля, не потеряйте их, перезаписав файлы.
Я залил модуль, но не вижу его в списке модулей
Скорее всего залили либо не то, либо не туда. Заливать нужно содержимое каталога /upload в корень сайта. Корневой каталог сайта - это тот, в котором лежит главный файл index.php, config.php и каталоги /admin, /catalog, /image, /system Самый простой способ проверить, что залито именно то и именно туда, это посмотреть наличие файла по следующему пути: /admin/controller/module/simple.php Если каталог админа переименован, то и проверять нужно по своему пути, подставив вместо admin название своего каталога админ части.
В каких языковых файлах хранятся тексты модуля?
Все названия полей, заголовков у наборов полей, тексты ошибок настраиваются в админке модуля на вкладке Поля. Весь "статический" текст (названия блоков, кнопок и т.п.) хранится в /catalog/language/russian/account/simpleregister.php /catalog/language/russian/checkout/simplecheckout.php
После установки модуля у меня перестало работать все/админ панель/сайт начал тормозить/появились ошибки на разных страницах
Модуль ничего не правит в опенкарте и не заменяет его системные файлы. Единственные правки: 1) В файле /system/library/url.php добавляется следующий код: 
        // SIMPLE START
        global $config;

        if (!empty($config) && method_exists($config, 'get') && $config->get('simple_settings')) {
            $get_route = isset($_GET['route']) ? $_GET['route'] : (isset($_GET['_route_']) ? $_GET['_route_'] : '');

            if ($config->get('simple_replace_cart') && $route == 'checkout/cart' && $get_route != 'checkout/cart') {
                $connection = 'SSL';
                $route = 'checkout/simplecheckout';
            }

            if ($config->get('simple_replace_checkout') && $route == 'checkout/checkout' && $get_route != 'checkout/checkout') {
                $route = 'checkout/simplecheckout';
            }

            if ($config->get('simple_replace_register') && $route == 'account/register' && $get_route != 'account/register') {
                $route = 'account/simpleregister';
            }

            if ($config->get('simple_replace_edit') && $route == 'account/edit' && $get_route != 'account/edit') {
                $route = 'account/simpleedit';
            }

            if ($config->get('simple_replace_address') && $route == 'account/address/update' && $get_route != 'account/address/update') {
                $route = 'account/simpleaddress/update';
            }

            if ($config->get('simple_replace_address') && $route == 'account/address/insert' && $get_route != 'account/address/insert') {
                $route = 'account/simpleaddress/insert';
            }
        }
        // SIMPLE END
Этот код подменяет ссылки на страницы модуля, если эта подмена включена в админ панели модуля. Этот код не влияет на работоспособность модуля и его можно удалть, страницы модуля в любом случае будут доступны по прямым ссылкам (т.е. если ввести их вручную в адресной строке браузера): /index.php?route=checkout/simplecheckout /index.php?route=account/simpleregister /index.php?route=account/simpleaddress /index.php?route=account/simpleedit Сами ссылки на страницы модуля можно прописать прямо в шаблоне Вашей темы. Также этот код не означает, что удаляются стандартные страницы регистрации и заказа, они также доступные по прямым ссылкам: /index.php?route=checkout/checkout /index.php?route=account/register 2) Вкмоды, которые ставятся вместе с модулем нужны только лишь для отображения кастомных полей модуля в админ части и в письмах. Они также не критичны для работы модуля и их можно удалить, при этом сами кастомные поля будут работать и сохраняться, просто не будут отображаться в админке. Значения полей можно получить, создав код для получения (смотри соответствующий раздел FAQа)
Я вижу пустую страницу, когда захожу на любую страницу модуля
Если видите пустую страницу в админ панели модуля, то откройте файл /admin/index.php Если же видите пустую страницу модуля в основной части сайта, то откройте файл /index.php Затем в этом файле сразу после строки <?php добавьте error_reporting(E_ALL); ini_set('display_errors', TRUE);
Я установил модуль, но ссылки регистрация и заказ ведут на стандартные страницы опенкарта
Проверьте по порядку 1) Залейте файл upload/system/library/url.php из архива модуля в каталог /system/library/ в вашем магазине. 2) В админ панели модуля в основных настройках страниц проверьте, что включена замена ссылок (по умолчанию они включены) и сохраните настройки Проверяйте ссылки с главной страницы опенкарта, а не обновляя стандартную регистрацию или заказ, модуль их не трогает и они остаются теми же самыми. Обычно первых двух шагов достаточно, если же уверены, что все залили и включили правильно, но ссылки не подменились, то еще раз проверьте в файле /system/library/url.php наличие кода между // SIMPLE START и // SIMPLE END Если ссылки все же остаются старыми, то скорее всего они жестко прописаны в Вашей теме, откройте каталог Вашей темы и выполните по файлам поиск строк checkout/checkout account/register Если найдете указанные строки, то сделай замену на checkout/simplecheckout account/simpleregister Иногда ссылки жестко прописаны в каких-нибудь модулях всплывающей корзины
Модуль отображается криво и не работает (не реагирует на кнопки и на изменения полей)
1) Убедитесь, что залили файлы из каталога /upload/catalog/view/theme/default/ в Вашу тему /catalog/view/theme/ВАШАТЕМА/ 2) Если файлы залиты, но стили и/или скрипты не подключаются, то скорее всего в Вашей теме вырезано подключение статики и скриптов в шаблоне. Откройте файл catalog/view/theme/ВАШАТЕМА/template/common/header.tpl Проверьте наличие следующих блоков и добавьте их если нужно до тэга </head> 
      <!-- подключение стилей модулей -->
      <?php foreach ($styles as $style) { ?>
      <link rel="<?php echo $style['rel']; ?>" type="text/css" href="<?php echo $style['href']; ?>" media="<?php echo $style['media']; ?>" />
      <?php } ?>
      <!-- подключение скриптов модулей -->
      <?php foreach ($scripts as $script) { ?>
      <script type="text/javascript" src="<?php echo $script; ?>"></script>
      <?php } ?>
Иногда не хватает либо блока для подключения стилей, либо для подключения джаваскриптов, но чаще всего вырезаны оба куска кода.
Модуль отображается не по всей ширине страницы или наоборот не совпадает с шириной шаблона
Откройте админку модуля на вкладке Интеграция - Тема и выполните шаги по интеграции с Вашей темой. Необходимо создать два файла simple_header.tpl и simple_footer.tpl и залить их в /catalog/view/theme/ВАШАТЕМА/template/common/ Для ряда тем в архиве модуля есть готовые шаблоны (в каталоге /templates)
Корзина в двухколоночном макете наезжает на другую колонку, что делать?
В админке модуля на вкладке "Страница заказа" - "Общие настройки" установите значения ширин колонок в процентах, подобрав их таким образом, чтобы устранить перекрытие колонок. Либо отключите отображение колонки "модель" в корзине модуля в настройках блока "Корзина". Как правило данная колонка редко используется, занимая при этом свободное пространство. Иногда общая ширина шаблона темы очень узкая сама по себе и вышеуказанные шаги не помогают. В таком случае рекомендуется изменить разметку блоков в админке модуля и отображать блок корзины во всю ширину страницы.
Где хранятся стили модуля?
Стили модуля определены в этом файле /catalog/view/theme/ВАШАТЕМА/stylesheet/simple.css
Текст миникорзины в шапке опенкарта на странице заказа отличается от текста на других страницах
Откройте файл /catalog/language/russian/checkout/cart.php найдите строку с текстом $_['text_items'] = '....'; и замените ею подобную строку в файле /catalog/language/russian/checkout/simplecheckout.php Некоторые темы содержат этот текст в других местах, например тема Coloring. Например для темы Coloring в файле \catalog\language\russian\checkout\simplecheckout.php в строке $_['text_items'] = '....'; укажите текст 
        <div class="products"><b>%s</b> товаров, </div><div class="prices">на <b>%s</b></div>
Как поместить поля покупателя в блок адреса?
Это сделать невозможно, так как блоки покупателя и блок адреса представляют собой разные модели опенкарта (и разные таблицы в базе данных со связью один ко многим). Единственная возможность - разместить блоки друг под другом и включить настройку "объединить с блоком покупателя" либо создать новые кастомные поля с привязкой к объекту адреса и добавить их в набор полей адреса. Поля Имя и Фамилия можно удалить либо в блоке покупателя, либо в блоке адреса, модуль будет подставлять их из другого блока (либо из покупателя в адрес, либо наоборот)
Я поменял стили модуля, но не вижу изменений
Нажмите в браузере CTRL+R, чтобы очистить кэш стилей. Редко в некоторых темах используется сжатие скриптов и стилей, необходимо почистить их кэш.
Где хранятся шаблоны полей модуля?
Это файлы, имя которых начинается с simple_row, хранящиеся в каталоге catalog/view/theme/ВАШАТЕМА/template/common/ либо в catalog/view/theme/default/template/common/ если забыли скопировать файлы шаблонов модуля в свою тему при установке
Я использую джаваскрипт плагин для улучшения вида полей, как включить его для полей модуля?
Вставьте необходимый вызов джаваскрипта прямо в админке модуля в разделе Интеграция - Джаваскрипт
Я использую модуль для сжатия стилей и джаваскриптов и модуль не работает
В разделе Интеграция - Джаваскрипт включите соответствующую настройку (Используется сжатие Minify).
Я использую тему Journal 2 и модуль конфликтует с ней (либо я использую другую тему со встроенным сжатием стилей и скриптов)
В разделе Интеграция - Джаваскрипт включите настройку "Используется сжатие Minify".
Как поменять ссылки на страницы модуля на красивые ЧПУ ссылки?
ЧПУ модули работают как "хуки" класса /system/library/url.php, обрабатывая вызов метода link(), через которые создаются все ссылки в опенкарте. Как правило данные модули работают по некой таблице соответствий вида: маршрут - короткая ссылка Логика работа данных модулей может отличаться, но в целом является одинаковой. Сами соответствия могут задаваться как в админке этих модулей, так и напрямую в таблицах соответствий в базе данных (вариант хранения соответствий может отличаться у разных модулей) То есть для модуля Simple нужно задать такие соответствия: checkout/simplecheckout - короткая ссылка на заказ account/simpleregister - короткая ссылка на регистрацию
Как добавить необходимые варианты доставки или оплаты?
Все варианты доставки и оплаты добавляются через установку и настройку соответствующих модулей доставки или оплаты.
Как изменить текст модулей доставки или оплаты?
Каждый модуль имеет свой языковой файл, в которым можно изменить любой текст. Файлы модулей доставки лежат тут /catalog/language/НУЖНЫЙ/shipping/ Файлы модулей оплаты лежат тут /catalog/language/НУЖНЫЙ/payment/
Блок доставки не отображается совсем (также как и блок адреса доставки)
Проверьте у ваших товаров настройку "требует доставки" В некоторых сборках опенкарта в классе корзины присутствует ошибка, проверьте, нет ли ее у Вас. Откройте файл /system/library/cart.php, найдите метод public function hasShipping() { и посмотрите, не стоит ли сразу за указанной строкой возврат false: return false; Если данная строка присутствует сразу после названия метода, то удалите ее.
Не отображаются варианты доставки или оплаты
Проверьте по порядку: 1) Если не отображаются варианты доставки, то проверьте у ваших товаров настройку "требует доставки" 2) Проверьте геозоны модулей доставки, установлены ли они или нет ("нет" означает, что выбрано "все регионы") 3) Если модули доставки/оплаты имеют непустые геозоны в своих настройках, а поля страны и/или региона при этом удалены с наборов полей адреса, то в основных настройках полей проверьте у полей "страна" и "регион" значения по умолчанию. Эти значения должны соответствовать геозоне модулей доставки/оплаты, чтобы эти модули могли отображаться. Например у модулей доставки выбрана геозона "Российская федерация", следовательно у поля "страна" нужно выбраться "Россия" в качестве значения по умолчанию. Если поле страны не удалено из набора полей, то задавать значение по умолчанию необязательно, так как покупатель может выбрать сам значение этого поля. 4) В настройках блоков оплаты или доставки проверьте настройку 'Не отображать указанный метод пока не заполнены поля адреса' 5) В настройках блоков оплаты или доставки проверьте настройки отображения для разных групп покупателя либо связи между модулями оплаты и доставки 6) Если ничего не помогает, то наберите в адресной строке браузера адрес стандартного заказа /index.php?route=checkout/checkout и проверьте отображение модулей там. Если они не отображаются нигде, то нужно еще раз тщательнее проверить настройки соответствующих модулей и написать авторам.
Не выбираются варианты доставки или оплаты
Те варианты доставки или оплаты, которые неактивны, являются "заглушками" модуля (задаются соответственно в настройках блоков доставки или оплаты) Это просто текст, который отображается вместо реальных вариантов, когда они не могут быть отражены в силу своих настроек, например модуль доставки или оплаты не возвращает информацию пока адрес пуст (когда например у него в настройках задана геозона). Заглушки нужны чтобы проинформировать пользователя о доступных вариантах доставки или опплаты, предложив, например, ввести свой адрес. Когда модуль доставки или оплаты возвратит информацию, то она будет отображена вместо заглушки и этот вариант станет активным.
Долго открывается страница заказа или долго перезагружаются блоки
Сам модуль не содержит в себе никакого кода, который мог бы долго выполняться. В силу своей специфики модуль вынужден загружать модули оплаты и доставки, некоторые из которых обращаются к удаленным серверам (например для расчета доставки) и это очень затратная по времени операция. Почему иногда приходится нажимать на кнопку "Оформить заказ" два раза?
В этом "виновата" логика опенкарта: перед загрузкой модуля оплаты вся информация о заказе должна быть сохранена в базе данных со специальным статусом 0. После этого управление передается модулю оплаты, который подтверждает заказ, переводя его статус в значение большее нуля. Таким образом первая кнопка "оформить заказ" сохраняет заказ в базе данных, а вторая кнопка "подтвердить заказ" является кнопкой модуля оплаты. Заказы со статусом ноль могут быть просмотрены в админке опенкарта, если установить фильтр заказов по статус "брошенный".
Как можно убрать вторую кнопку "подтвердить заказ"?
В модуле для этого реализованы следующие возможности: 1) "Показывать форму оплаты как можно раньше" - отвечает за то, что при каждой аякс перезагрузке формы (например при изменении количества) происходит попытка сохранения заказа в базе данных со статусом 0 (создание "заготовки" заказа), данное сохранение возможно только если все поля формы являются "валидными", то есть поля проходят все правила проверки без ошибок. В случае сохранения заказа в БД также подгружается форма оплаты и пользователю остается кликнуть по кнопке "подтвердить заказ". Данная настройка разделена на две части "показывать форму оплаты как можно раньше для гостей" и "показывать форму оплаты как можно раньше для зарегистрированных пользователей" для гибкости настройки. Внимание! Если включена данная настройка и пользователь зарегистрирован, то при любом заходе покупателя на страницу заказа вся информация о заказе сразу же будет сохранена в БД со статусом 0 и будет подгружена форма оплаты (так как все поля загружаются из аккаунта пользователя и являются "валидными"). Но это совершенно не значит, что пользователь хочет сделать заказ, поэтому если увидите заказ среди "брошенных" в админке модуля, то это совершенно ничего не означает. Отключить данное поведение можно, как уже было сказано, в админке модуля. 2) Модуль пытается сам нажать кнопку "подтвердить заказ" в случае, если форма оплаты не содержит никакого текста или полей ввода. В редких случаях ошибки в сторонних джаваскриптах могут останавливать этот процесс и приходится нажимать вручную. В таком случае нужно открыть консоль разработчика (F12) и посмотреть есть ли ошибки. 3) Можно переделать платежный модуль, перенеся его форму на шаг вперед, до нажатия на кнопку "оформить заказ". Я создал несколько модификаций для ряда популярных модулей оплаты, например для "банковсого перевода", "paypal pro" и нескольких других.

Почему при создании/подтверждении заказа надпись "подождите, происходит переход к оплате" подвисает на несколько секунд/минут?
Данный этап не зависит от модуля, ту же самую картину можно наблюдать и со стандартным заказом. В этот момент происходит подтверждение заказа модулем оплаты, то есть модуль оплаты вызывает метод confirm() модели заказа опенкарта (catalog/model/checkout/order.php) В этом методе происходит отправка почты (и смс, если конечно же установлено такое расширение) и это самая медленная операция на данном этапе, скорость выполнения которой зависит от метода отправки почты (в случае варианта "sendmail" также зависит и от хостинга). Попробуйте сменить метод отправки в основных настройках магазина на "smtp" или наоборот на "sendmail". В случае медленной работы почты при способе "sendmail" можно попробовать написать письмо в техподдержку хостинга. Также иногда "тормозит" перенаправление на страницу оплаты у модуля "paypal", но с этим ничего нельзя поделать.
Как отключить текст "подождите, происходит переход к оплате"?
Отображение данного текстка можно отключить в админ панели модуля на вкладке "страница регистрации" - "общие настройки"
Заказ оформлен, оплачен покупателем, но не виден в админке. Почему?
Данная проблема не зависит от варианта оформления заказа и имеет общий характер, в чем легко убедиться, сделав заказ через стандартную страницу заказа, набрав в адресной строке браузера ссылку на стандартный заказ (/index.php?route=checkout/checkout) либо отключив модуль Simple. Как уже было написано в ответе на вопрос про двойное нажатие кнопки, перед загрузкой формы оплаты заказ сохраняется в базе данных со статусом 0 и по умолчанию данный заказ не виден в админке опенкарта. После загрузки формы оплаты все дальнейшие действия по подтверждению заказа (т.е. действия по изменению статуса заказа) выполняет модуль оплаты. Все онлайн способы оплаты (вебмани, робокасса, пейпал и так далее) используют так называемые "колбэки" ("обратные вызовы") для подтверждения заказа. Т.е. это специальные ссылки в вашем магазине, обращаясь к которым платежный шлюз может подтвердить оплату (а значит и заказ) в вашем магазине. Если подтверждения не происходит, то нужно проверить по шагам: 1) Настройки данных ссылок в админке модулей оплаты, иногда они отображаются как текст для копирования и последующей вставки в настройках аккаунта в платежной системе. 2) Настройки данных ссылок в личном кабинете платежной системы. 3) Доступны ли ссылки извне 4) Нет ли ошбок в логах опенкарта или в логах сервера 5) Ряду модулей оплаты необходим curl для выполнения запросов, проверьте, установлен и доступен ли он на Вашем сервере. 5) Если все шаги пройдены, то стоит написать автору модуля оплаты.
Как включить или отключить регистрацию при заказе?
В настройка полей у поля регистрация установите "да" или "нет" значением по умолчанию. После этого можно удалить поле из набора полей в блоке покупателя, будет использоваться значение, установленное по умолчанию.

Как отключить перезагрузку формы при изменении полей?
В основных настройках полей есть настройка "влияет ли данное поле на методы оплаты, доставки или на общую сумму заказа". Отключите ее, чтобы не вызывать перезагрузку формы.
У поля город стоит галочка для перезагруки блока, но при изменении поля перезагрузки не происходит
Отключите автозаполенение по полю города в общих настройках страницы заказа.
Как убрать ненужные страны и/или регионы из списков выбора?
В основном меню опенкарта в разделе "Локализация" откройте "Страны" и удалите ненужные страны, аналогичным образом можно поступить и с регионами. Для подстраховки после удаления можете очистить кэш опенкарта, удалив все файлы в каталоге /system/cache
Что означает привязка к объекту в настройках полей?
Это означает объект, вместе с которым будет храниться поле (а значит и загружаться). Поле покупателя хранится в таблице customer_simple_fields, привязано к покупателю по его идентификатору и загружается вместе с основной информацией о покупателе. Поле адреса хранится в таблице address_simple_fields, привязано к адресу по его идентификатору и загружается вместе с информацией об адресе. Поле заказа хранится в таблице order_simple_fields, привязано к заказу по его идентификатору и загружается вместе с информацией о заказе. Внимательно обдумывайте вариант привязки поля при создании новых кастомных полей, чтобы избежать вопросов типа: почему поле не заполнено, когда пользователь зарегистрирован и начинает оформлять заказ.

Что за раздел "модули" в админке модуля и для чего нужны группы настроек страницы заказа?
Начиная с версии 4.0 в модуле можно создавать наборы (группы) настроек для страницы заказа для использования в различных сценариях. Например, можно создать минималистичный набор настроек и отображать модуль на любой странице опенкарта как модуль в изначальном смысле этот слова (т.е. как часть страницы в верхней, нижней, левой или правой частях). Также можно подключить только скрипты модуля и загружать модуль в попапе, используя к примеру fancybox или colorbox, прописав в нужном месте ссылку на страницу заказ с указанием нужной группы настроек в качестве параметра, например: <a class="fancybox" href="index.php?route=checkout/simplecheckout&group=1" alt="Оформить заказ">Оформить заказ</a> Точно также в попапе можно открыть страницу регистрации.

Проверьте по порядку: 1) Таблицы залиты и их префиксы приведены в соответствие с префиксом Вашей базы 2) Включена настройка для определения адреса по IP в админ панели модуля в настройках нужных страниц 3) Почистите кэш опенкарта, удалив все файлы в каталоге /system/cache 4) Попросите друга открыть страницу модуля у себя, потому что Вашего IP адреса может не быть в базе 5) Очень редко настройки сервера таковы, что не передают данные о посетителе в массиве $_SERVER и надо задать вопрос техподдержке.
Почему мой адрес не определяется по IP или определяется неверно?
База данных была собрана из открытых источников, поэтому не обладает полнотой и точностью. В любом случае, даже платная база не может гарантировать Вам 100% определение и точность. Вы всегда можете попробовать сменить вариант определения адреса в настройках модуля, а также реализовать собственный вариант определения адреса через АПИ модуля в файле /catalog/model/tool/simplegeo.php

1) В настройках поля капчи в настройках полей на вкладке валидация уберите проверку по длине (по очевидным причинам) 2) Добавьте поле в набор нужных полей, сделайте его обязательным (это важно) 3) Откройте файл \catalog\controller\account\simpleregister.php и после строки public function index($args = null) { добавьте $this->config->set('config_google_captcha_status', true); $this->config->set('config_google_captcha_public', 'публичный ключ'); $this->config->set('config_google_captcha_secret', 'секретный ключ'); по аналогии со странице и файлом checkout/simplecheckout.php
